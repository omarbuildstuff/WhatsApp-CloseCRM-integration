Create `src/whatsapp/messageHandler.ts` with these three exports:

**1. `extractBody(msg: proto.IWebMessageInfo): string | null`** -- Pure function, no I/O.

Extract text from WhatsApp message by type, in this priority order:
- `msg.message.conversation` -- plain text
- `msg.message.extendedTextMessage?.text` -- link previews, replies
- `msg.message.imageMessage?.caption` -- image with caption
- `msg.message.videoMessage?.caption` -- video with caption
- `msg.message.documentMessage?.caption` -- document with caption
- `msg.message.documentWithCaptionMessage?.message?.documentMessage?.caption` -- wrapped document
- `msg.message.locationMessage` -- format as `"Location: {lat}, {lng}"`
- `msg.message.contactMessage?.displayName` -- format as `"Contact: {displayName}"`
- For audio: return `"[Audio message]"` (SYNC-05 placeholder -- Close API rejects empty string)
- For sticker: return `"[Sticker]"`
- All other/unknown types: return `null`

**2. `detectMediaType(msg: proto.IWebMessageInfo): string | null`** -- Pure function.

Return the media type string based on which message property exists:
- `imageMessage` -> `"image"`
- `videoMessage` -> `"video"`
- `audioMessage` -> `"audio"`
- `documentMessage` or `documentWithCaptionMessage` -> `"document"`
- `stickerMessage` -> `"sticker"`
- `locationMessage` -> `"location"`
- `contactMessage` -> `"contact"`
- Otherwise -> `null`

**3. `MessageHandler` class** with a single public method `async handle(repId: string, msg: proto.IWebMessageInfo): Promise<void>`.

The handle method implements this exact flow:

```
1. FILTER CHAIN (return early, no side effects):
   - if (!msg.key.remoteJid) return
   - if (msg.key.remoteJid.endsWith('@g.us')) return     // skip groups
   - if (msg.key.fromMe) return                           // skip outbound
   - if (!msg.message) return                              // skip system stubs
   - if (!msg.key.id) return                               // skip messages with no dedup key

2. EXTRACT:
   - const jid = msg.key.remoteJid
   - const waMessageId = msg.key.id
   - const body = extractBody(msg)
   - const mediaType = detectMediaType(msg)
   - const e164 = normalizeJidToE164(jid)
   - const tsSec = Number(msg.messageTimestamp ?? 0)
   - const timestamp = new Date(tsSec * 1000)              // PITFALL: seconds, not ms!

3. LEAD LOOKUP:
   - const lead = e164 ? await phoneCache.lookup(e164) : null

4. PERSIST TO DB (ALWAYS -- regardless of lead match -- SYNC-03):
   INSERT INTO messages (id, rep_id, direction, wa_jid, phone_e164, lead_id, body, media_type, timestamp)
   VALUES ($1, $2, 'inbound', $3, $4, $5, $6, $7, $8)
   ON CONFLICT (id) DO NOTHING                             // SYNC-04 dedup
   Parameters: [waMessageId, repId, jid, e164 ?? null, lead?.leadId ?? null, body, mediaType, timestamp]
   Check: const inserted = (result.rowCount ?? 0) > 0

5. SYNC TO CLOSE (only if lead matched AND row was inserted -- not a duplicate):
   if (lead && inserted) {
     const activityId = await closeClient.postWhatsAppActivity({
       lead_id: lead.leadId,
       direction: 'inbound',
       external_whatsapp_message_id: waMessageId,
       message_markdown: body ?? '',                        // fallback empty string for Close API
     });
     if (activityId) {
       UPDATE messages SET close_activity_id = $1 WHERE id = $2
       Parameters: [activityId, waMessageId]
     }
   }
```

Wrap steps 3-5 in try/catch with `logger.error({ repId, waMessageId, err }, 'Failed to process inbound message')`. The filter chain (step 1) and extraction (step 2) should NOT be wrapped -- errors there should propagate.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WhatsApp -> MessageHandler | Untrusted message content from external WhatsApp users crosses into the system |
| MessageHandler -> PostgreSQL | Message body text (user-controlled) written to database |
| MessageHandler -> Close API | Message body text (user-controlled) sent to external API as activity content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | MessageHandler.persistMessage | mitigate | All SQL uses parameterized queries ($1, $2, ...) -- message body never interpolated into SQL strings. Established pattern from Phase 1-2. |
| T-03-02 | Tampering | MessageHandler.handle | mitigate | lead_id comes only from phoneCache.lookup() (trusted path via Close API search), never from message content. Attacker cannot control which lead a message is assigned to. |
| T-03-03 | Denial of Service | MessageHandler.handle | mitigate | msg.key.fromMe filter prevents outbound echo loops. Group chat filter (@g.us) prevents high-volume group message floods. Both filters run before any I/O. |
| T-03-04 | Information Disclosure | logger calls | accept | Message body logged at error level only (on failure). Normal processing does not log message content. Acceptable for MVP debugging. |
| T-03-05 | Spoofing | Close API postWhatsAppActivity | mitigate | Close API auth uses existing Basic auth config (API key as username). No new auth surface introduced. axiosRetry already configured. |
| T-03-06 | Repudiation | MessageHandler.persistMessage | mitigate | Every message persisted to PostgreSQL with timestamp, rep_id, and wa_message_id before any Close API call. DB is the audit trail. |
</threat_model>
