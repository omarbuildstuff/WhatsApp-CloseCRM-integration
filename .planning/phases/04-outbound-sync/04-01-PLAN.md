Create `src/close/webhookHandler.ts` as a new file. This is the core outbound sync handler implementing a 5-step pipeline.

**Imports:**
- `createHmac`, `timingSafeEqual` from `crypto` (Node built-in)
- `Request`, `Response` from `express`
- `pool` from `../db/pool`
- `sessionManager` from `../whatsapp/sessionManager`
- `closeClient` from `./client`
- `config` from `../config`
- `jidEncode` from `@whiskeysockets/baileys`
- `pino` for logger
- `CloseWebhookActivityData` from `./types`

**Export a single async function:**
```typescript
export async function handleCloseWebhook(req: Request, res: Response): Promise<void>
```

**Step 1: HMAC-SHA256 Signature Verification**
- Extract `close-sig-hash` and `close-sig-timestamp` headers from the request
- `req.body` will be a raw `Buffer` (route-level `express.raw` middleware is set in Plan 02)
- Compute HMAC: `data = sigTimestamp + rawBody.toString('utf8')`, key = `Buffer.from(config.closeWebhookSecret, 'hex')`, algorithm = `sha256`
- Compare computed digest (hex) with `sigHash` using `timingSafeEqual` on Buffer representations
- If headers missing or verification fails: respond `403 { error: 'Invalid signature' }` and return
- Add replay protection: reject if `close-sig-timestamp` is more than 5 minutes old (`Math.abs(Date.now() - parseInt(sigTimestamp) * 1000) > 300_000`). NOTE: timestamp may be in seconds or milliseconds -- check both. If `parseInt(sigTimestamp) > 1e12` treat as ms, otherwise treat as seconds and multiply by 1000.
- Create a helper function `verifySignature(rawBody: Buffer, sigTimestamp: string, sigHash: string, secret: string): boolean` that encapsulates the HMAC logic including the `timingSafeEqual` comparison. This function must catch exceptions (e.g., mismatched buffer lengths) and return `false`.

**Step 2: Parse JSON payload**
- `JSON.parse(rawBody.toString('utf8'))` wrapped in try/catch
- On parse failure: respond `400 { error: 'Invalid JSON' }` and return
- Extract `data` with fallback: `const data: CloseWebhookActivityData = parsed?.event?.data ?? parsed?.data;`
  This handles both assumed nesting patterns until live verification confirms one.
- If `!data?.id`: respond `200 { ok: true }` (unrecognized event shape, drop silently) and return

**Step 3: Loop Guard -- CRITICAL, NON-NEGOTIABLE (per OUT-02, CLAUDE.md)**
- `if (data.external_whatsapp_message_id)` -- this field is non-null when OUR integration created the activity
- Log at debug level: `'Loop guard: dropping webhook for our own activity'` with `{ activityId: data.id, externalWaId: data.external_whatsapp_message_id }`
- Respond `200 { ok: true }` and return immediately
- This MUST be the FIRST business logic check after parsing, before any other processing

**Step 4: Direction filter**
- `if (data.direction !== 'outbound')` -- only process outbound activities created by a rep in Close UI
- Respond `200 { ok: true }` and return
- Log at debug level

**Step 5: Respond 200 early, then process async**
- `res.status(200).json({ ok: true })` -- acknowledge the webhook immediately to prevent Close timeout retries
- All remaining logic runs after the response is sent. Errors are logged, not returned to Close.

**Step 6: Rep routing (async, after 200 response)**
- Query: `SELECT id FROM reps WHERE close_user_id = $1 AND status = 'connected'` with `[data.user_id]`
- If no rows: log warning `'No connected rep for Close user'` with `{ closeUserId: data.user_id }` and return
- Get `repId` from `rows[0].id`
- Get socket: `sessionManager.getSession(repId)`
- If no socket: log warning `'No active Baileys session'` with `{ repId }` and return

**Step 7: Resolve customer phone number (async)**
- Call `closeClient.getLeadContacts(data.lead_id)` to get the customer's E.164 phone
- If null: log warning `'Cannot resolve phone for lead'` with `{ leadId: data.lead_id }` and return
- Strip leading `+`: `const digits = phoneE164.replace(/^\+/, '')`
- Build JID: `jidEncode(digits, 's.whatsapp.net')` (from Baileys -- do NOT hand-roll string concat)

**Step 8: Send via Baileys (async)**
- `const result = await sock.sendMessage(jid, { text: data.message_markdown })`
- `const waMessageId = result?.key?.id ?? null` -- handle undefined return from Baileys
- If `waMessageId` is null: log error `'sendMessage returned undefined'` with `{ repId, closeActivityId: data.id }` and return

**Step 9: Persist outbound message to PostgreSQL (async)**
- INSERT into `messages` table:
  ```sql
  INSERT INTO messages (id, rep_id, direction, wa_jid, phone_e164, lead_id, close_activity_id, body, timestamp)
  VALUES ($1, $2, 'outgoing', $3, $4, $5, $6, $7, NOW())
  ON CONFLICT (id) DO NOTHING
  ```
  Parameters: `[waMessageId, repId, jid, phoneE164, data.lead_id, data.id, data.message_markdown]`
- `ON CONFLICT DO NOTHING` handles the edge case of duplicate webhook delivery

**Step 10: Update Close activity with real WA message ID (async)**
- `await closeClient.updateWhatsAppActivity(data.id, { external_whatsapp_message_id: waMessageId })`
- This sets the external ID so future webhooks for this activity will trigger the loop guard
- Log info: `'Outbound message sent and Close activity updated'` with `{ repId, waMessageId, closeActivityId: data.id }`

**Wrap steps 6-10 in a single try/catch.** On error: log at error level with `{ err, closeActivityId: data.id }`, message `'Outbound webhook processing failed'`. Do NOT rethrow -- the 200 was already sent.

**IMPORTANT:** Always return 200 to Close once the request is authenticated (step 1 passes). Returning 500 causes Close to retry, which can lead to duplicate sends. The only non-200 responses are 403 (bad signature) and 400 (invalid JSON).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Internet -> /webhook/close | Untrusted HTTP POST from Close (or attacker) crosses into our server |
| Our server -> WhatsApp | Outbound message send to customer's phone via Baileys |
| Our server -> Close API | PATCH/PUT to update activity with WA message ID |
| Our server -> PostgreSQL | Insert outbound message record |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | POST /webhook/close | mitigate | HMAC-SHA256 signature verification using close-sig-hash + close-sig-timestamp headers with crypto.timingSafeEqual in verifySignature() |
| T-04-02 | Tampering | HMAC comparison | mitigate | Use crypto.timingSafeEqual, never === or indexOf for signature comparison -- prevents timing attacks |
| T-04-03 | Tampering | Replay attack | mitigate | Reject requests where close-sig-timestamp is older than 5 minutes from server time |
| T-04-04 | Elevation of Privilege | Infinite send loop | mitigate | Loop guard: if data.external_whatsapp_message_id is non-null, return 200 immediately -- this is the #1 architecture rule per CLAUDE.md |
| T-04-05 | Denial of Service | Close webhook retries | mitigate | Always return 200 after authentication passes; never return 5xx for business logic failures. Errors are logged internally. |
| T-04-06 | Spoofing | Rep routing | mitigate | Query reps WHERE close_user_id = $1 AND status = 'connected' -- only active reps with matching Close user ID can send |
| T-04-07 | Information Disclosure | Error responses | mitigate | 403 returns generic "Invalid signature", never reveals HMAC details. Error details logged server-side only. |
| T-04-08 | Tampering | SQL injection | mitigate | All database queries use parameterized $1/$2 placeholders, never string interpolation |
| T-04-09 | Tampering | Raw body mutation | mitigate | express.raw() route-level middleware preserves Buffer for HMAC before JSON.parse (registered before express.json() in Plan 02) |
</threat_model>
